# BOSS 薪资显示修复设计

## 目标

安全模式下导入用户当前 BOSS 列表页中已经可见的薪资，使数据库和项目前端显示正确的月薪区间，而不是 `0-0K`。

## 根因

BOSS 薪资节点的文本使用 Unicode 私有区字符表示数字。页面借助字体渲染为正常数字，但扩展从 `textContent` 读取到的是 `U+E031` 到 `U+E03A`。现有 `parseSalary` 只接受 ASCII 数字，因此将真实薪资当成未知值落库。

## 方案

1. 在扩展的纯提取器中增加小型 `decodeSalary` 函数，将 `U+E031..U+E03A` 转换为 `0..9`，其他字符原样保留。
2. 提取时先解码再传给本地 bridge；bridge 和后端现有合约不变。
3. `parseSalary` 再做一层同样的解码，防止旧扩展或其他入口传入未解码字符。
4. 完整的解码后文本作为 `salaryText` 通过 bridge、Spring Boot 导入合约、MySQL 和查询 VO 保留，兼容 `K`、`元/天`和 `·13薪` 等真实格式。
5. 前端优先显示 `salaryText`；旧数据仍根据 `salaryMin/salaryMax` 显示月薪，两者都无值时显示“薪资面议”。

## 边界

- 只读当前用户已打开页面上可见的岗位卡片。
- 不调用隐藏接口，不处理验证码，不降低或规避平台风控。
- 不新增依赖，不改变课程微服务架构。

## 验收

- 扩展测试中 `\uE032\uE036-\uE033\uE031K` 被提取为 `15-20K`。
- 归一化测试中同一输入落为 `15000` 和 `20000`。
- `\uE033\uE031\uE031-\uE037\uE031\uE031元/天` 保留并显示为 `200-600元/天`，不换算成虚假月薪。
- 重新导入后，已有 BOSS 岗位按 `externalId` 更新，不新增重复数据。
- 项目全量测试、前端构建和真实页面展示均通过。
