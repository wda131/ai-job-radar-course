# AI 求职雷达必选技术栈完善设计

日期：2026-07-14

## 目标与边界

在不扩大求职雷达核心业务范围的前提下，补齐课程要求的 Redis 与 RabbitMQ，并把现有五个业务域整理为六个可由六名成员独立承担的前后端功能模块。保留现有登录、职位、匹配、收藏投递、模拟面试功能，不引入可选 Elasticsearch 和 Docker，降低答辩现场依赖数量与演示风险。

## 推荐架构

- 前端：Vue 3、Vue Router、Axios、Vite（Node.js 构建与运行）。
- 入口：Spring Cloud Gateway，统一转发 `/api/**`。
- 注册发现：Nacos，全部服务注册到课程环境。
- 服务调用：OpenFeign，用于匹配、投递、面试服务查询职位或用户信息。
- 持久化：MySQL 8 + MyBatis Plus。
- 缓存：职位服务使用 Redis 缓存职位列表和职位详情，统一 10 分钟 TTL。
- 异步消息：投递服务在新投递和状态更新后发送 RabbitMQ 事件；新增通知服务异步消费并落库。
- 服务：user、job、match、application、interview、notification 六个业务微服务，加 gateway 与 common。

## 核心消息链路

1. 用户创建投递或更新投递状态。
2. application-service 完成数据库写入。
3. 服务向 `radar.application.exchange` 直连交换机发送 `application.status.changed` 事件。
4. notification-service 监听 `radar.notification.queue`，按事件编号幂等写入通知表。
5. Vue 消息中心通过 Gateway 查询消息、显示未读状态并支持单条或全部已读。
6. RabbitMQ 短暂不可用时，主投递流程不回滚，服务记录明确日志；答辩环境启动 RabbitMQ 后展示完整异步链路。

## Redis 设计

- `radar:jobs::search:*`：职位分页与筛选结果。
- `radar:jobs::detail:*`：职位详情。
- 采用 Spring Cache 与 `RedisCacheManager`，JSON 序列化、10 分钟过期。
- 职位为课程演示中的只读样例数据，不增加管理后台和复杂缓存失效功能。

## API 与数据

- `GET /api/notifications`：当前用户消息列表。
- `GET /api/notifications/unread-count`：未读数量。
- `PUT /api/notifications/{id}/read`：标记单条已读。
- `PUT /api/notifications/read-all`：全部已读。
- 新增 `notification` 表，包含 event_id、user_id、type、title、content、read_status、created_at；event_id 唯一保证重复消费安全。

## 六人功能分工

1. 成员一（队长）：用户与个人档案模块，负责登录鉴权、档案维护和对应页面。
2. 成员二：职位雷达模块，负责职位检索、详情、Redis 缓存和对应页面。
3. 成员三：智能匹配模块，负责画像匹配、OpenFeign 调用、匹配报告和对应页面。
4. 成员四：收藏投递模块，负责收藏、投递进度、RabbitMQ 事件发布和对应页面。
5. 成员五：模拟面试模块，负责题目、作答、评分、面试记录和对应页面。
6. 成员六：消息中心模块，负责 RabbitMQ 消费、通知入库、已读管理和对应页面。

姓名尚未在代码仓库中提供，PPT 使用“成员一至成员六”的可编辑占位，并明确标注答辩前替换真实姓名；不臆造组员信息。

## 演示与验收

- 一键启动脚本检查 MySQL、Nacos、Redis、RabbitMQ 后启动全部后端和前端。
- 演示顺序：登录 → 职位筛选 → 收藏/投递 → 查看异步通知 → 智能匹配 → 四题模拟面试 → 更新投递状态 → 再次查看通知。
- 验收包含 Maven 测试、Node 单元测试、前端生产构建、关键 API 冒烟测试、Redis 命中证据、RabbitMQ 消费与通知落库证据。
- PPT 展示课程前三天技术路线、系统架构、六个业务模块、Redis/RabbitMQ 链路、Git 提交截图、测试结果、六人工作内容和现场演示脚本。
