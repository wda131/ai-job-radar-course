# 职位操作与模拟面试交互修复设计

## 目标

修复课程版中三个影响演示完整性的交互问题：模拟面试题不可直观看到、收藏状态没有反馈、投递文案与业务状态不一致。

## 推荐方案

1. 职位页加载职位时，同时读取当前用户的收藏与投递列表，以岗位 ID 维护页面状态。收藏按钮在“收藏”和“已收藏”之间切换；已收藏使用高亮样式并支持取消收藏。
2. 投递按钮统一使用“投递”文案。创建投递后，后端直接写入 `APPLIED` 状态和投递时间；前端显示“已投递”并禁用重复操作。
3. 创建模拟面试后直接跳转至该面试会话。面试页也会在没有指定会话时自动打开最新记录，并在答题区上方完整列出四道题，当前题和已完成题使用不同状态样式。

## 数据流

- `GET /api/favorites` 与 `GET /api/applications` 为职位卡片提供初始状态。
- 收藏操作继续复用现有新增、取消接口，不增加后端表或接口。
- `POST /api/applications` 返回状态为 `APPLIED` 的记录。
- `POST /api/interviews` 返回会话 ID，前端跳转到 `/interviews?session=<id>` 并加载详情。
- 后端统一将 `Long` 序列化为字符串，避免 MyBatis-Plus 雪花 ID 超出 JavaScript 安全整数范围；前端全程按字符串匹配会话 ID。

## 验收标准

- 刷新职位页后，已收藏和已投递状态仍能恢复。
- 收藏按钮文案、颜色和 `aria-pressed` 与真实收藏状态一致。
- 投递成功后按钮立即显示“已投递”，后端看板进入“已投递”列。
- 创建面试后自动进入对应会话，四道题全部可见，仍按顺序逐题提交并显示反馈。
- 超长会话 ID 在响应、URL 和详情请求中保持完全一致。
- 不新增课程外依赖，不修改既有数据库结构。

## 测试

- 后端单元测试验证新投递直接进入 `APPLIED` 且记录投递时间。
- 前端使用 Node 内置测试验证收藏、投递按钮状态映射和面试会话选择逻辑。
- 公共模块测试验证 Java `Long` ID 输出为 JSON 字符串。
- 完整 Maven 测试、前端测试与 Vite 构建通过后，再进行浏览器端真实操作验收。
