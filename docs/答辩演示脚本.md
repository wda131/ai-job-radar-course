# AI 求职雷达答辩演示脚本

## 答辩前必须替换

- 将 PPT 中“成员一（队长）～成员六”替换为六位成员真实姓名。
- 六位成员分别使用姓名拼音 GitHub 账号完成至少 2 次有效提交。
- 队长重新截取 GitHub 提交记录，替换 PPT 第 13 页截图。
- 在答辩电脑执行 `scripts/check-environment.ps1`，确认必需环境。

## 六人业务模块分工

| 成员 | 业务模块 | 端到端工作 | 答辩职责 |
| --- | --- | --- | --- |
| 成员一（队长） | 用户与画像 | 登录、JWT、个人资料、求职偏好、相关测试 | 开场、需求、总体架构、联调统筹 |
| 成员二 | 职位雷达 | 职位检索、筛选、详情、Redis 缓存、相关测试 | 演示职位检索与 Redis key |
| 成员三 | 智能匹配 | 画像评分、职位匹配、OpenFeign 调用、相关测试 | 讲解匹配规则与服务调用 |
| 成员四 | 求职行动 | 收藏、投递、状态管理、RabbitMQ 发布、相关测试 | 演示收藏与投递状态 |
| 成员五 | 模拟面试 | 创建面试、四题切换、答案提交、评分、相关测试 | 演示四题面试流程 |
| 成员六 | 消息中心 | RabbitMQ 消费、通知落库、未读与已读、相关测试 | 演示消息中心与异步通知 |

## 六分钟现场演示

1. `00:00–01:00`：成员一登录系统，说明所有请求从 Vue 3 经 Gateway 进入 Nacos 注册的微服务。
2. `01:00–02:00`：成员二搜索职位并打开详情，连续查询后执行 `redis-cli keys "*"`，展示 `radar:jobs:*` 缓存。
3. `02:00–03:00`：成员四点击收藏与投递，指出收藏按钮状态变化、投递按钮变为“已投递”。
4. `03:00–04:00`：成员六进入消息中心，展示投递事件产生的通知、未读标识、单条已读和全部已读。
5. `04:00–05:00`：成员五创建模拟面试，依次切换题目 1–4，并填写答案。
6. `05:00–06:00`：成员三或队长展示测试结果、GitHub 提交记录和六人模块分工。

## 启动顺序

```powershell
# 1. 检查环境
.\scripts\start-rabbitmq.ps1
.\scripts\check-environment.ps1

# 2. 导入数据库（首次运行）
mysql --default-character-set=utf8mb4 -uroot -proot < .\sql\ai_job_radar.sql

# 3. 启动后端
.\scripts\start-backend.ps1

# 4. 启动前端
cd .\ai-job-radar-web
npm install
npm run dev
```

浏览器打开 Vite 控制台输出的地址（通常是 `http://127.0.0.1:5173`，端口占用时会自动顺延）。演示账号为 `student`，密码为 `123456`。

## 现场风险与降级顺序

1. RabbitMQ 必须优先启动；如果临时不可用，投递发布器会容错，核心投递流程仍可演示，但异步通知改用 PPT 中的代码与测试证据说明。
2. Redis 不可用时职位查询会失败，因此答辩前必须用 `redis-cli ping` 检查并看到 `PONG`。
3. Nacos 启动后等待服务全部注册，再打开前端；不要在注册未完成时开始演示。
4. 保留 PPT 第 7 页的三张系统截图，作为网络或现场环境故障时的展示备份。

## 常见答辩问题

- 为什么使用 Redis：职位列表和详情属于高频读数据，缓存能减少重复数据库查询，设置 10 分钟 TTL 控制陈旧数据。
- 为什么使用 RabbitMQ：投递是核心业务，通知是后续动作；异步事件能解耦两个服务，并允许通知独立扩展。
- 为什么使用 Gateway：统一路由和跨域配置，前端只访问一个入口。
- 为什么使用 Nacos：服务实例动态注册发现，Gateway 和 OpenFeign 不需要写死业务服务地址。
- 为什么没有 Elasticsearch 和 Docker：两者为可选技术；本项目优先保证必选技术完整进入业务主链路和现场稳定性。
